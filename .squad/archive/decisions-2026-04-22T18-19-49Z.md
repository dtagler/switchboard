## 2026-04-20: CRITICAL ARCHITECTURAL FINDING — WH_KEYBOARD_LL cannot identify source device

**By:** owner (prior-art research). **Status:** Invalidates the prior "hook-blocks-only-built-in-keys" architecture. Triggers a v2 revision.

### The finding

`WH_KEYBOARD_LL`'s `KBDLLHOOKSTRUCT` carries `vkCode`, `scanCode`, `flags`, `time`, `dwExtraInfo` — but **no device handle**. There is no reliable way to know which keyboard sent a key from inside the low-level hook. Therefore "hook installed; block built-in events; pass external events through" is not implementable with `WH_KEYBOARD_LL` alone — the hook can't tell the difference, so it would block the Nuphy too.

### The three real architectures (per Microsoft docs + community consensus)

1. **Raw Input API (`WM_INPUT`)** — `RAWINPUTHEADER.hDevice` *does* identify the source device. But Raw Input only delivers events to the registered window; it cannot block input system-wide. **Usable for detection, not blocking.**

2. **Keyboard filter driver (kernel-mode)** — selective per-device blocking at the input stack. Requires WDK, signed driver, BSOD risk. **Out of scope** for a portable single-exe.

3. **Disable the internal keyboard device via SetupAPI / PnP** — equivalent to Device Manager → right-click → Disable. APIs: `SetupDiSetClassInstallParams` + `DIF_PROPERTYCHANGE` with `DICS_DISABLE` / `DICS_ENABLE`. **Requires admin elevation.** Used by every working open-source project in this space. **This is the standard, proven approach.**

### Prior art (all use approach #3)

- `amrsohil/Automatic-Laptop-Keyboard-Disabler` — Python, USB-focused, requires Python + admin
- `snifer/KeyboardLocker` — manual hotkey toggle
- `sylveon/key-disable` — per-device disable
- DevCon + AutoHotkey — common scripted pattern (AHK watches device events; DevCon disables/enables by hardware ID)
- PS/2 registry method — only works on PS/2; Surface uses HID-over-I2C, so this approach is dead. Skip.

### Cross-compile validated

`cargo-zigbuild --target aarch64-pc-windows-msvc --release` is the standard and works inside Docker. Jerry's Docker plan stands.

### Implications (Jerry to address in arch v2)

1. **Drop hook-as-blocker model.** New model: Nuphy connects → disable internal keyboard device via SetupAPI; Nuphy disconnects → re-enable it. The low-level hook is no longer the blocking mechanism.
2. **Admin elevation is now required.** Conflicts with "no admin if avoidable" preference. Options to evaluate: manifest-requested elevation (UAC every launch); tiny Windows service running SYSTEM with IPC; scheduled task with highest privileges; just accept the UAC prompt. Jerry recommends one.
3. **Hardware ID lookup needed.** Owner will run read-only `Get-PnpDevice` / `Get-CimInstance` commands on the host. Jerry produces the exact command list.
4. **Fail-safe must NOT depend on the internal keyboard.** Internal keyboard is fully off at the OS level — no key on it can trigger anything. Fail-safe channels remaining: external keyboard (Nuphy — defeats purpose if Nuphy is the failure), mouse/tray click, touchpad, touchscreen, power button, panic file. George + Newman re-evaluate; Jerry picks primary + backup.

### What this preserves from prior decisions

- Stack: Rust + windows-rs + tray-icon (still correct; nothing here is hook-specific)
- Docker-only dev rule (reaffirmed; toolchain still containerized)
- Module structure (still ~6 modules, with `hook` module repurposed or removed and replaced by `device-controller` for SetupAPI calls)

## 2026-04-20: Jerry's architecture proposal (PENDING OWNER REVIEW)

**Source:** Jerry (Lead / Windows Architect), architecture pass, 2026-04-20.

### Stack Recommendation

**Recommendation: Rust** (windows-rs + tray-icon + tokio)

Rust is the correct choice for this project. One paragraph:

Rust provides native ARM64 Windows compilation via `aarch64-pc-windows-msvc`, produces a single static executable (~3–5 MB for a tray app), has first-class Win32 and WinRT bindings through Microsoft's own `windows-rs` crate, cross-compiles cleanly from Linux Docker containers using `cross` or Zig as linker, and imposes near-zero runtime overhead. The ownership model eliminates entire classes of memory safety bugs in the keyboard hook — exactly where we can't afford crashes. The tradeoff is a steeper learning curve than C# or Python, but the safety and size constraints make that trade worthwhile.

### Comparison Table

| Criterion | Python + PyInstaller | Rust | C# / .NET 8 NativeAOT | Go | C++ (Win32/WinRT) |
|-----------|---------------------|------|----------------------|-----|-------------------|
| **ARM64 native exe** | ❌ Experimental, no mature cross-compile | ✅ Tier 1 target | ⚠️ Works, cross-compile from Linux fragile | ✅ Native support | ✅ Native |
| **Single exe** | ✅ (but huge) | ✅ | ✅ | ✅ | ✅ |
| **Docker cross-compile** | ❌ Needs ARM64 Python + bootloader, painful | ✅ `cross` or Zig linker, well-documented | ⚠️ Possible but needs MinGW-w64 ARM64 toolchain, less tested | ✅ Trivial | ⚠️ Needs MSVC cross-toolchain |
| **Win32 hook story** | ⚠️ ctypes works, less tested on ARM64, GIL concerns | ✅ windows-rs, direct FFI, no GC | ✅ P/Invoke, mature | ✅ golang.org/x/sys/windows | ✅ Native |
| **WinRT BT story** | ⚠️ winrt package exists, ARM64 wheels available but less battle-tested | ✅ windows-rs includes full WinRT projection, Microsoft-maintained | ✅ CsWinRT, mature | ⚠️ go-winrt exists but experimental, gaps in coverage | ✅ Native C++/WinRT |
| **Binary size** | 45–70 MB | 3–5 MB | 3–6 MB | 8–12 MB | 1–2 MB |
| **Idle RAM** | 30–50 MB (Python runtime) | 2–5 MB | 5–10 MB | 8–15 MB | 2–4 MB |
| **Dev velocity** | High (if it works) | Medium | High | High | Low |

### Why Rust Wins

1. **ARM64 story is solid.** `aarch64-pc-windows-msvc` is a Tier 1 target. PyInstaller ARM64 cross-compile is not production-ready in 2026.
2. **Binary size meets "lightweight" constraint.** 3–5 MB vs. 50+ MB for Python.
3. **Win32 + WinRT in one crate.** `windows-rs` is maintained by Microsoft; both `SetWindowsHookEx` and `Windows.Devices.Enumeration.DeviceWatcher` are first-class.
4. **Docker cross-compile works.** The `cross` tool or Zig-as-linker approach is documented and functional.
5. **Memory safety where it matters.** The keyboard hook callback runs on a system thread with strict timing. Rust's compile-time guarantees beat Python's GIL and runtime exceptions.

### Honest Tradeoffs

- **Learning curve.** Rust is harder than Python or C#. Newman, Kramer, and George need to be comfortable with ownership, lifetimes, and async.
- **Compile times.** Full rebuilds take longer than Go or C#. Incremental builds are fast.
- **Ecosystem maturity for tray apps.** `tray-icon` is community-maintained, not Microsoft-blessed. Acceptable but we should pin versions carefully.
- **No hot reload.** Unlike Python, you can't tweak and re-run instantly. Mitigated by fast incremental builds.

### High-Level Architecture

#### Module List

| Module | Responsibility |
|--------|---------------|
| **core** | Application entry point, single-instance enforcement, startup/shutdown orchestration |
| **hook** | `SetWindowsHookEx WH_KEYBOARD_LL` lifecycle — install, callback dispatch, unhook. Owns fail-safe timer. |
| **bluetooth** | WinRT `DeviceWatcher` for Nuphy Air75 detection. Exposes `is_connected: bool` signal. |
| **tray** | System tray icon, context menu (Enable/Disable, Quit), tooltip state |
| **failsafe** | Escape-hold detection, persistent disable flag, hook release trigger |
| **config** | Load/save settings (e.g., target device ID, enabled state). File-based, no registry. |

#### Threading Model

```
┌─────────────────────────────────────────────────────────────────┐
│                         MAIN THREAD                             │
│  - Win32 message pump (GetMessage loop)                         │
│  - Tray icon event handling                                     │
│  - Hook callback dispatch (WH_KEYBOARD_LL runs here)            │
│  - Fail-safe timer check (on each key event)                    │
└─────────────────────────────────────────────────────────────────┘
            │
            │ channel (mpsc)
            ▼
┌─────────────────────────────────────────────────────────────────┐
│                     ASYNC RUNTIME THREAD (tokio)                │
│  - WinRT DeviceWatcher event loop                               │
│  - Sends BT state changes to main thread via channel            │
└─────────────────────────────────────────────────────────────────┘
```

**Key points:**
- The keyboard hook callback **must** run on the thread that called `SetWindowsHookEx` and has a message pump. This is the main thread.
- WinRT async operations run on a tokio runtime in a background thread. We do NOT block the hook callback waiting on BT.
- Communication is one-way: BT thread → main thread via `std::sync::mpsc` channel. Main thread reads channel on each message pump iteration.

#### Process Lifecycle

1. **Startup**
   - Check single-instance mutex (`CreateMutexW`). If already running, signal existing instance and exit.
   - Load config. If persistent-disable flag is set, show tray icon in "disabled" state; do not install hook.
   - Install keyboard hook.
   - Start DeviceWatcher for target BT device.
   - Enter message pump.

2. **Runtime**
   - BT watcher fires Added/Removed events → sends `Connected`/`Disconnected` to main thread.
   - Main thread updates internal state. When Nuphy connected → hook suppresses built-in keyboard. When disconnected → hook passes through.
   - Tray menu "Enable/Disable" toggles suppression without removing hook.
   - Fail-safe runs on every key-down: if Escape held 10s, trigger emergency disable.

3. **Clean Shutdown** (user clicks Quit)
   - Unhook keyboard hook.
   - Stop DeviceWatcher.
   - Release mutex.
   - Exit process.

4. **Crash Behavior**
   - If process crashes, Windows automatically unhooks `WH_KEYBOARD_LL`. Built-in keyboard resumes.
   - Persistent-disable flag survives crash (it's file-based). On next launch, app starts disabled if flag is set.

#### BT Signal Contract (Kramer → Newman)

**Signal shape:**
```rust
enum BluetoothState {
    Connected,
    Disconnected,
    Unknown, // watcher not yet started or errored
}
```

**Delivery:** Via `mpsc::Sender<BluetoothState>` from bluetooth module to hook module.

**Debounce:** Bluetooth module debounces internally. It waits 500ms after a state change before sending, to avoid rapid connect/disconnect chatter. Hook receives only stable states.

**If watcher dies:**
- Bluetooth module sends `Unknown` and logs error.
- Hook treats `Unknown` as "pass through" — does NOT suppress keyboard.
- Fail-safe remains operational regardless.

**Hook never blocks on BT.** The channel is checked non-blocking via `try_recv()` on each message pump iteration.

#### §1. Trigger Definitions

**1.1. PRIMARY TRIGGER: Tray Icon Left-Click**

- A single left-click (`WM_LBUTTONUP`) on the tray icon toggles the hook state.
- If currently enabled → disable. If currently disabled → enable.
- No confirmation dialog. Instant toggle.

**1.2. BACKUP TRIGGER: Registry Watchdog**

- A dedicated watchdog thread polls `HKCU\Software\BluetoothKeyboardBlocker\EmergencyDisable` (REG_DWORD) every 2 seconds.
- If value is non-zero, trigger fail-safe.
- A bundled utility `emergency-disable.exe` sets this key and exits.

**1.3. CONVENIENCE TRIGGER: Escape Held 10 Seconds**

- Inside the `WH_KEYBOARD_LL` callback, track Escape key-down time.
- If held for 10 continuous seconds without key-up, trigger fail-safe.
- Key-repeat events do NOT reset the timer.
- Other keys pressed while Escape is held do NOT cancel the timer.
- This trigger depends on the hook callback being responsive. If the hook is hung, this will not fire.

**1.4. IMPLICIT SAFETY: Startup Grace Period**

- On app launch, wait 5 seconds before installing the keyboard hook.
- During this window, the tray icon is visible and interactive.
- User can quit or disable the app before the hook activates.

#### §2. Effect (All Triggers)

When any trigger fires, the following MUST happen in this order:

**2.1. Unhook immediately.** Call `UnhookWindowsHookEx` to remove the keyboard hook. This restores normal keyboard input.

**2.2. Set persistent disable flag.** Write a flag to config file (`disabled = true`) or registry. This survives app restart and system reboot.

**2.3. Update tray icon.** Change icon to "disabled" state (gray or ⊘ overlay). Update tooltip to "Keyboard hook disabled."

**2.4. Log the event.** Write timestamp, trigger source ("TRAY_CLICK" / "WATCHDOG" / "ESCAPE_HOLD"), and "FAILSAFE_TRIGGERED" to log file.

**2.5. Do NOT exit the app.** Tray icon remains visible so user can re-enable or quit.

#### §3. Independence

**3.1. PRIMARY (Tray Click) Independence:**
- Does NOT depend on Bluetooth watcher, hook callback, or async runtime.
- DOES depend on message pump being responsive and tray icon being visible.

**3.2. BACKUP (Registry Watchdog) Independence:**
- Does NOT depend on message pump, hook callback, Bluetooth watcher, or async runtime.
- The watchdog thread is self-contained: spawned at startup, polls registry, calls `UnhookWindowsHookEx` directly.
- Can unhook even if main thread is deadlocked.

**3.3. CONVENIENCE (Escape-10s) Independence:**
- DOES depend on hook callback executing and receiving key events.
- If hook callback is hung, this trigger will not fire — user must use PRIMARY or BACKUP.

**3.4. IMPLICIT (Grace Period) Independence:**
- Complete independence — hook does not exist yet during grace period.

#### §4. Watchdog Contract

**4.1.** The watchdog thread MUST be spawned at app startup, before the hook is installed.

**4.2.** The watchdog thread MUST poll the registry key every 2 seconds (±500ms tolerance).

**4.3.** On detecting trigger, the watchdog MUST:
1. Call `UnhookWindowsHookEx` with the stored hook handle.
2. Verify the hook is removed (handle invalid or test key-through).
3. If unhook fails, call `ExitProcess(1)` — a dead app cannot hold a hook.
4. Write persistent disable flag.
5. Reset registry key to 0.
6. Post message to main thread: "update tray icon to disabled" (best effort — main thread may be hung).

**4.4.** The watchdog thread MUST NOT block on any resource owned by the main thread (no mutexes shared with hook callback).

#### §5. Recovery

**5.1.** After any fail-safe triggers, user re-enables via:
- Right-click tray icon → "Enable" menu item, OR
- Left-click tray icon (toggles back to enabled), OR
- Quit and restart app, then enable via tray menu.

**5.2.** The "Enable" action MUST:
- Clear the persistent disable flag
- Clear the registry emergency key (if set)
- Re-install the keyboard hook
- Update tray icon to "enabled" state

**5.3.** There is no automatic re-enable. User must take explicit action.

#### §6. Testability

**6.1. Unit Tests (run in Docker, no Windows):**
- Timer starts on Escape down, cancels on Escape up before 10s
- Timer does NOT reset on key-repeat
- Timer does NOT cancel on other keys
- Watchdog thread polling logic (mock registry)

**6.2. Integration Tests (run on ARM64 Windows):**
- Tray left-click disables hook, built-in keyboard works immediately
- Escape-hold 10s disables hook (when hook is responsive)
- Registry key + watchdog disables hook
- Grace period: hook not active for first 5 seconds
- Persistent flag is written and honored on restart

**6.3. Stress Tests (run on Surface Laptop 7):**
- Add `Sleep(50)` to hook callback, verify tray click still works
- Add `Sleep(50)` to hook callback, verify Escape-hold still triggers
- Deadlock main thread (simulate), verify watchdog still unhooks

**6.4. Manual Test Script:**
```
PRIMARY TRIGGER (Tray Click):
1. Start app, verify hook enabled (built-in keyboard suppressed when Nuphy connected)
2. Left-click tray icon
3. Verify: tray icon changes to "disabled," built-in keyboard works, log shows TRAY_CLICK

BACKUP TRIGGER (Registry Watchdog):
1. Start app with hook enabled
2. Run: reg add HKCU\Software\BluetoothKeyboardBlocker /v EmergencyDisable /t REG_DWORD /d 1 /f
3. Wait ≤3 seconds
4. Verify: hook disabled, tray icon changed, log shows WATCHDOG
5. Verify: registry key reset to 0

CONVENIENCE TRIGGER (Escape Hold):

2. Hold Escape for 10 seconds
3. Verify: hook disabled, tray icon changed, log shows ESCAPE_HOLD

### §A. Core Mechanism (NEW)

- **When Nuphy connects** → call SetupAPI `SetupDiSetClassInstallParams` + `SetupDiCallClassInstaller(DIF_PROPERTYCHANGE)` with `DICS_DISABLE` on the internal keyboard device. The internal keyboard is **fully disabled at the OS level** — it doesn't exist to any app, lock screen, or login.
- **When Nuphy disconnects** → same APIs with `DICS_ENABLE`. Internal keyboard restored.
- The keyboard hook (`WH_KEYBOARD_LL`) is **no longer the blocking mechanism** and never blocks/suppresses keys. See §C for what happens to it.

### §B. Admin Elevation Strategy — Decision: Option (c) Scheduled Task

**Decision: Scheduled task with highest privileges, triggered at user logon.** Jerry leads, George concurs, Newman dissents (prefers option (a) manifest UAC for simplicity).

| Option | Verdict | Why |
|--------|---------|-----|
| (a) Manifest `requireAdministrator` | ❌ Rejected | UAC every launch breaks autostart cleanly (elevated apps can't be Run-key autostarted), defeats "lightweight, unobtrusive." George flagged: standard-user accounts cannot run it at all. |
| (b) Windows service + named pipe | ❌ Rejected | Violates the owner's zero-install constraint. Service install/uninstall, SCM interaction, registry writes — not portable. |
| **(c) Scheduled task at logon, highest privileges** | ✅ **Selected** | Preserves zero-install runtime (no MSI, no service), zero UAC at launch after one-time setup, runs elevated automatically. Setup script (`setup-task.ps1`) is the only one-time admin step. George notes the ~30-second logon delay before the task fires is actually a *safety feature* — natural grace period where the internal keyboard is guaranteed working. |
| (d) COM elevation / RunAs caching / hybrid | ❌ Rejected | Either equivalent to (b) with extra steps, or a security disaster (cached elevated tokens). |

**Tradeoff acknowledged (Newman + George):** Option (c) is not literally zero-install — there's a one-time `setup-task.ps1` run that needs admin to register the scheduled task. The runtime experience is zero-install (no MSI, no service, just the .exe), but bootstrapping requires one elevated PowerShell session. Owner accepts this in exchange for no UAC at every launch.

**Setup flow:**
1. Ship `main.exe` + `setup-task.ps1` + `recovery.exe` (see §G) in a single folder.
2. User runs `setup-task.ps1` once (UAC prompt) → script creates scheduled task pointing at `main.exe`, trigger = user logon, run with highest privileges.
3. Task survives reboots; main.exe launches elevated automatically every logon.
4. To uninstall: tray menu → "Uninstall (Safe Removal)" — re-enables internal keyboard, removes scheduled task, exits.

### §C. Hook Decision

**Decision: Drop the hook entirely.** Newman recommended dropping it; Jerry initially proposed keeping it for Escape-10s. Owner makes the final call but the team's leaning is to cut it:
- Hook can no longer block (can't differentiate source device — confirmed by the WH_KEYBOARD_LL finding).
- Hook's only remaining job was Escape-10s convenience. With internal keyboard fully disabled, Escape-10s only works from the *Nuphy* anyway — and if the Nuphy is the failure source, this trigger is useless.
- Removing the hook eliminates a P/Invoke boundary, simplifies threading, reduces attack surface.
- Layers 0/1/2 below are sufficient without it.

**Optional alternative if owner wants a panic key:** `RegisterHotKey(MOD_WIN | MOD_NOREPEAT, VK_ESCAPE)` for `Win+Escape` → re-enable internal keyboard. No hook required. Decision deferred to owner.

### §D. Module Structure (7 modules)

| Module | Responsibility |
|--------|---------------|
| `core` | App lifecycle, main thread, message pump, tray event loop, owns all state decisions |
| `device_controller` (NEW) | SetupAPI calls (`SetupDiGetClassDevs`, `SetupDiEnumDeviceInfo`, `SetupDiSetClassInstallParams`, `SetupDiCallClassInstaller`). Stateless: re-scans device tree on every call. **Verify-after-call** required (read back `CM_Get_DevNode_Status` to confirm intended state). Only the main thread invokes this module. |
| `bluetooth` | Nuphy connect/disconnect watcher (WinRT `DeviceWatcher` or `WM_DEVICECHANGE`). Debounced 500ms. Sends signals to `core`. |
| `tray` | System tray icon, left-click = panic re-enable (always re-enables, no questions asked), right-click menu (Uninstall/Quit/Force Re-enable). |
| `failsafe` | Registry-watchdog thread (Layer 2). Independent of main thread. |
| `deadman` (NEW per George) | Raw Input subscriber on Nuphy device handle. If internal keyboard is disabled and no Nuphy keypress for 60s, force re-enable internal keyboard. Mitigates Bluetooth-stack-stall lockout (George scenario #7). |
| `config` | Persistent flags, hardware-ID cache (with re-enumeration fallback). |

Plus a separate companion binary: **`recovery.exe`** (see §G) — the break-glass tool.

### §E. Process Lifecycle

**Startup (CRITICAL — crash recovery primitive):**
1. App launches elevated via the scheduled task.
2. **§0 INVARIANT — FIRST EXECUTABLE CODE:** Call `device_controller::enable_internal_keyboard()` unconditionally, before tray init, before BT watcher, before reading config. **Re-enable first. Evaluate second.**
3. **Verify** the call succeeded by reading back `CM_Get_DevNode_Status`. If it failed, retry once at 500ms; if still failed, enter **safe mode**: tray notification "Cannot re-enable internal keyboard — manual recovery required" (link to §G), set `internal_keyboard_disable_forbidden = true`, never attempt disable this session.
4. Acquire single-instance mutex (`CreateMutexW`). If another instance is running, exit cleanly. (George scenario #10 — multi-instance race on SetupAPI.)
5. Spawn registry watchdog thread.
6. Init tray icon.
7. Start BT watcher + dead-man timer.
8. Wait 5-second grace period.
9. Evaluate Nuphy state. If connected → disable internal keyboard. If not → leave it enabled.
10. Enter message loop. Begin writing heartbeat file every 5s.

**Runtime:** Nuphy connect/disconnect → toggle internal keyboard via `device_controller`. Tray left-click is always "panic re-enable." All state changes are **verify-after-call**.

**Clean shutdown:** Re-enable internal keyboard (defensive, even if we think it's enabled). Release mutex. Exit.

**Crash behavior — explicit:** If the process dies while internal keyboard is disabled, the device stays disabled across reboots — `DICS_DISABLE` is a registry change in `HKLM\SYSTEM\CurrentControlSet\Enum`, persistent. Mitigations stacked:
- §0 invariant on next launch (auto-recovers because scheduled task auto-launches every logon)
- Dead-man timer if Nuphy keypress activity drops (during crash, no app = no detection, but the next reboot's §0 invariant covers this)
- `recovery.exe` break-glass tool if scheduled task is broken

### §F. Fail-safe v2 — Layered Defense

| Layer | Mechanism | Recovers from | Requires | Status |
|-------|-----------|---------------|----------|--------|
| **0** | **Always-re-enable-on-start invariant** (§E step 2) | App crash, force-kill, BSOD, hard power-off | App restart (auto via scheduled task on logon) | **NEW — primary crash recovery** |
| **1** | **Tray left-click = panic re-enable** | App is running but in bad state; user-driven | Pointer input (mouse/touchpad/touchscreen), admin token | Mandatory. ALWAYS re-enables, never checks state. |
| **2** | **Registry watchdog thread** — polls `HKCU\Software\BluetoothKeyboardBlocker\EmergencyDisable` every 2s; on non-zero, calls `DICS_ENABLE`, verifies, resets key, posts message to main thread | Main-thread deadlock or hang | App running, admin | Mandatory. Companion `emergency-enable.exe` (renamed from v1's `emergency-disable.exe`) sets the registry key. |
| **D** | **Dead-man's switch** (Raw Input on Nuphy device) — if internal kb is disabled AND no Nuphy keypress for 60s, force re-enable | Bluetooth stack stall — Nuphy disconnect signal never fires (George scenario #7) | App running, Nuphy device handle obtainable via Raw Input | **NEW — required by George.** Trade-off: brief double-keyboard window when user is idle 60+s, then types on Nuphy. Acceptable. |
| **3** | **Manual recovery path** — Device Manager via touchscreen / on-screen keyboard / Surface BIOS keyboard fail-safe (Vol-Up + Power) / Safe Mode | All software failures | User action; touchscreen or another keyboard | Documented in README and in `recovery.exe` UI. |

**Triggers v2:**
- **PRIMARY: Tray left-click** — instant, always re-enables, no Bluetooth check, no flag check. The "panic button."
- **BACKUP: Registry watchdog** — survives main-thread deadlock; does NOT survive app crash (Layer 0 covers that).
- **CONVENIENCE: Escape-10s** — ❌ DROPPED unless owner asks for it back. Hook removed (§C).
- **IMPLICIT: 5-second startup grace period** — internal keyboard guaranteed enabled during this window.

**Effect of any trigger** (atomic, in this order):
1. Call `DICS_ENABLE` on internal keyboard.
2. **Verify** with `CM_Get_DevNode_Status`. If mismatch, retry once, then `ExitProcess(0)` so §0 invariant recovers.
3. Set persistent flag `internal_keyboard_enabled = true` (survives restart).
4. Update tray icon to "auto-disable inactive" state.
5. Log event with trigger source.
6. Do NOT exit the app (tray remains, user can re-engage auto-disable manually). Re-engagement requires explicit user action — no automatic revert.

### §G. `recovery.exe` Break-Glass Tool (NEW per George)

A separate, minimal companion binary (~100 KB, no dependencies on main app's state). **Only job:** re-enable internal keyboard.

- User keeps it in a folder (or pinned to taskbar). If the main app is broken/uninstalled/locked-out scenario, double-click → UAC → re-enables all HID-class keyboard devices in `Disabled` state → exits.
- Required because: scenarios where main app is dead (crashed + scheduled task broken, or user deleted main.exe without uninstall) leave the user locked out (George scenarios #2, #3, #6).
- Documented prominently in README ("If you are locked out, use `recovery.exe`").

### §L. Two new lockout classes v2 did not fully defend

**CLASS 1 — Lock-screen lockout.** User locks screen with Nuphy connected (internal keyboard disabled). Nuphy disconnects (auto-power-off, battery dies, user steps away with the keyboard). Internal keyboard remains disabled. User returns, can't unlock. v2 had no defense for this — the §0 startup invariant only fires on app start, the dead-man's switch can't see Raw Input from the secure desktop, and tray click obviously requires being already logged in.

**CLASS 2 — Cold boot lockout.** SetupAPI's disable persists via registry (K8). On next boot, internal keyboard is disabled before login screen renders. The user-context "At logon" scheduled task does NOT fire until AFTER credentials are entered (K10). If the Nuphy is dead/missing/charging at that moment, user is locked out at the login screen. **Worst case**: overnight Windows Update reboot. User wakes up, Nuphy in bag, locked out.

### §M. v3 defensive additions

**Layer 0B — Boot Recovery Task (NEW, primary for Class 2).**
- Second scheduled task created by `setup-task.ps1`. Trigger: "At system startup". Principal: SYSTEM. Action: `recovery.exe --boot-reenable`.
- Per K9: fires after PnP, before login screen. Per K12: SYSTEM has the privileges to call `DICS_ENABLE`.
- Behavior: re-enable internal HID-over-I2C keyboard unconditionally. The user-context tray app, on logon, will re-disable if Nuphy is connected — net cost is one short enable→disable cycle on every boot, totally invisible to the user.
- **Owner cost:** one extra scheduled task. No new install — `setup-task.ps1` creates both tasks together.

**Layer 0C — Shutdown Hook (NEW, defense-in-depth for Class 2).**
- Tray app handles `WM_QUERYENDSESSION` / `WM_ENDSESSION`. On clean shutdown/restart, unconditionally re-enable internal keyboard before letting the OS proceed.
- Per K13/K14: 20 s timeout is plenty for one SetupAPI call; covers all clean shutdown paths but NOT BSOD/power-loss (those still rely on Layer 0B).
- Why both 0B and 0C: 0C reduces unnecessary enable→disable cycles when shutdown was clean (most boots come back with no work for 0B); 0B catches everything else.

**Layer 0D — Lock-screen Re-enable (NEW, primary for Class 1).**
- Tray app calls `WTSRegisterSessionNotification` (per K7). On `WTS_SESSION_LOCK`: unconditionally re-enable internal keyboard. On `WTS_SESSION_UNLOCK`: if Nuphy is currently connected (verify via DeviceWatcher state), re-disable.
- **UX trade-off:** internal keyboard works on the lock screen even when the user might prefer it didn't. Owner accepted "safety over UX" already in v2 — this stays consistent.
- Edge: if app is OOM-killed during lock, no re-enable on unlock happens. Mitigated by Layer 0E.

**Layer 0E — Unlock Trigger on Scheduled Task (NEW, gap-fill from K11).**
- Modify the existing user-context scheduled task: add a second trigger "On workstation unlock". Set "If the task fails, restart every 1 minute, attempt 3 times". This guarantees the tray app is running after every unlock, not just initial logon.

**Layer §E-R2 — Reactive Re-enable on Nuphy Disconnect (CRITICAL — Kramer's finding).**
- Use **WinRT DeviceWatcher** (NOT WM_DEVICECHANGE — see correction below). On `Removed` event matching the Nuphy's container ID: immediately call `DICS_ENABLE` on the internal keyboard.
- Defends against Nuphy auto-power-off (K17) — the #1 real-world lockout vector. User notices ~500 ms of no input instead of being stuck until they discover OSK.
- Works while session is locked (K4) and unlocked alike. This is the single most important addition in v3.

### §N. Correction to v2

**v2 specified `WM_DEVICECHANGE` as the BT-signal mechanism. Newman verified this is wrong for our use case (K3): the message is not delivered to user-session apps using `DEVICE_NOTIFY_WINDOW_HANDLE` while the session is locked.**

**v3 mandate: use WinRT `Windows.Devices.Enumeration.DeviceWatcher` exclusively for BT presence detection.** This is async, service-driven, and unaffected by lock state. The `bluetooth` module spec in v2 §C should be updated accordingly. The `device_controller` module's BT input remains the same — only the signal source changes.

### §T. Path A vs Path B — decision and reasoning

**The reviewers' alternative — "use Raw Input with `RIDEV_INPUTSINK | RIDEV_NOLEGACY` to intercept and discard internal-keyboard events" — does not work.**

`RIDEV_NOLEGACY` is **per-process scope**, not system-wide. It suppresses legacy `WM_KEYDOWN`/`WM_KEYUP` messages **only in the registering process's message queue**. Other applications (Notepad, browser, etc.) continue to receive legacy keyboard messages from the internal keyboard normally. Path B as a *blocker* is impossible.

> **Microsoft Learn — RegisterRawInputDevices:** "If RIDEV_NOLEGACY is set, predefined keyboard messages are not posted to the message queue **for the application that registered the device**." (https://learn.microsoft.com/en-us/windows/win32/api/winuser/nf-winuser-registerrawinputdevices)

This was confirmed by independent web verification on 2026-04-21. Newman's spike design assumed global suppression and proposed `SendInput` re-injection of Nuphy events — that design solves a problem that doesn't exist (no global suppression to compensate for) and doesn't solve the actual problem (other apps still see internal-keyboard input).

**Path B has one valid use:** as an *observer* of which device sourced a given keystroke. Useful for telemetry or for confirming Path A's targeting (a debug build can register Raw Input and log every event's `hDevice` to verify the SetupAPI disable actually stopped events from the targeted device). Not a blocker.

**Verdict: v4 is Path A (SetupAPI device-disable), done correctly this time.**

### §V. v4 architecture — five layers, ordered

**Path A. Single user-mode process. No service. No driver. No scheduled task.** Admin elevation required (UAC manifest). Self-elevates on launch via ShellExecute "runas" if not elevated. Persists no state.

| # | Layer | Mechanism | When it fires |
|---|-------|-----------|---------------|
| 0 | **Re-enable on shutdown** | `WM_QUERYENDSESSION` and `WM_ENDSESSION`: synchronous SetupAPI `DICS_ENABLE` on the targeted keyboard child node. Block return until enable confirmed. | Logoff, restart, shutdown, hybrid shutdown (Fast Startup) |
| 1 | **Cold-start invariant** | On process launch: (a) elevate if needed, (b) SetupAPI `DICS_ENABLE` unconditionally on targeted node (defensive, in case any prior crash left it disabled), (c) WinRT query Nuphy state via `BluetoothDevice.FromIdAsync().ConnectionStatus` — NOT cached — (d) only if Nuphy confirmed connected, disable. | Every process launch (including autostart) |
| 2 | **Power-state hook** | `WM_POWERBROADCAST`: `PBT_APMSUSPEND` → re-enable synchronously before suspend. `PBT_APMRESUMEAUTOMATIC` → re-query Nuphy state from scratch; disable only if connected. Replaces v3's wrong `WTS_SESSION_LOCK` hook. | Sleep, hibernate, Modern Standby S0ix entry/exit |
| 3 | **Composite-device safety check** | Before any `DICS_DISABLE` call: enumerate target node's `ContainerId`; refuse if matches any non-keyboard device's `ContainerId` (esp. touchpad). Hard fail with logged reason. | Every disable attempt |
| 4 | **Tray-click manual override** | Click tray icon → toggle. Always available. Touchpad survives (Layer 3 guarantee). | User action |
| 5 | **Reactive Nuphy disconnect** | `DeviceWatcher.Updated` watching `System.Devices.Aep.IsConnected`. On `IsConnected = false` → re-enable internal keyboard within ~50ms. (NOT `Removed` — `Removed` only fires on unpair.) Polled fallback every 5s via `BluetoothDevice.FromIdAsync().ConnectionStatus` for Modern Standby resume edge cases. | Nuphy power-off, out-of-range, BT radio off |

**That's it. Five layers.** No 0B (boot task), no 0C (separate shutdown hook beyond Layer 0), no 0D (lock re-enable), no 0E (unlock task), no dead-man's switch, no registry watchdog, no heartbeat file, no recovery.exe.

### §Y. Scope-narrowing — honest

The original "never lock out" claim **cannot be honored under Path A** without external USB keyboard prereq. The unavoidable residuals:

- **BitLocker recovery prompts.** Triggered by Surface monthly firmware updates. Pre-OS context. Internal keyboard state at last shutdown is what BitLocker sees. **Mitigated** by Layer 0 (every shutdown re-enables) but not eliminated — if the app crashes mid-disable and the OS is force-rebooted before Layer 0 fires, internal keyboard stays disabled into BitLocker.
- **UEFI / boot manager / WinRE.** Same class of pre-OS risk.
- **Forced reboot from a disabled state** (BSOD, hard reset).

**v4 scope claim is downgraded to:**

> **"Safe during normal logged-in Windows sessions. Internal keyboard is restored before every shutdown, sleep, hibernate, lid-close, and on Nuphy disconnect. Pre-OS recovery (BitLocker, UEFI, WinRE) requires either a successful clean shutdown OR an external USB keyboard as documented prerequisite."**

Owner must accept this language. If not acceptable, the alternative is to **require external USB keyboard documented as part of normal use** (recommended pairing: bring USB keyboard to any work that might require monthly Surface firmware update / BitLocker prompt).

### §Z. Multi-user / FUS / RDP policy — explicit

SetupAPI device disable is machine-wide. Per-user app cannot safely disable a device that affects other sessions.

**v4 policy:**
1. On launch and on `WM_WTSSESSION_CHANGE` (`WTS_SESSION_LOGON` / `WTS_CONSOLE_CONNECT` / `WTS_REMOTE_CONNECT`): call `WTSEnumerateSessions` and count active interactive sessions.
2. **If > 1 active session:** refuse to disable. If currently disabled, re-enable immediately. Tray icon shows "paused — multi-session active." Resume normal behavior when single-session restored.
3. **No support for RDP / FUS / multi-user host configurations.** Documented as out of scope. README warns.

This is non-negotiable for safety. If owner wants multi-session support, the architecture must move to a per-session model (Path B observability + custom blocking via filter driver), which is several orders of magnitude more complex and outside scope.

## §AE. NEW Layer 3.5 — InstanceId Cache Invalidation & Re-Resolution (Newman)

HardwareId-based **identification** is durable; the **InstanceId** that SetupAPI returns from a match is per-PnP-enumeration and can change across SAM-bus surprise-removal, Modern Standby resume, firmware updates, and feature updates. The cache must be invalidated explicitly.

**Invalidation triggers (mandatory):**
1. Process start (cold-start invariant — Layer 1 always re-resolves)
2. `WM_DEVICECHANGE` matching SAM bus class GUID or our target's Parent path
3. `PBT_APMRESUMEAUTOMATIC` (Modern Standby / S3 / hibernate resume)
4. Shutdown handler entry (Layer 0 — always re-enumerate inside `WM_QUERYENDSESSION`; do not trust cached InstanceId)
5. Any SetupAPI call returning `ERROR_NO_SUCH_DEVINST` → invalidate, re-resolve once, retry

**Re-resolution discipline:** "Re-resolve" means re-run the §AC predicate against `SetupDiGetClassDevs(GUID_DEVCLASS_KEYBOARD, …)` from scratch — not lookup by cached InstanceId.

**Multi-match safety (I2):** If §AC predicate matches >1 device on any re-resolution, FAIL CLOSED — refuse to disable, log "duplicate match" with both candidates' full property dumps, surface in tray notification. Do NOT pick one.

**State-verification discipline (I3):** After every `DICS_DISABLE`, call `CM_Get_DevNode_Status` to verify the device's `ConfigFlags` reflect the requested state. If still enabled → log error, do not assume success. After every `DICS_ENABLE`, same.

**Layer 0 invariant (I7):** The shutdown handler MUST re-enumerate fresh inside `WM_QUERYENDSESSION` — cached InstanceId may be stale; the cost of re-enumeration is sub-100ms and acceptable inside the WaitToKillAppTimeout window (default 20s).

Full invariants table I1–I7 + race-condition analysis lives in `.squad/agents/newman/v4-deep-dive/` (Scribe to move from inbox).

### Fixes Applied

#### Blockers (4)

**Fix 1 — Sanity timer defeats resume_pending (Claude)**  
The 20 s sanity timer was calling `apply_policy()` unconditionally, which would re-disable the internal keyboard at the lock screen within 20 s of lid-open — defeating the careful resume-pending separation.

**Change:** Timer now gates on BOTH `!resume_pending` AND `WTSQuerySessionInformation(WTS_CURRENT_SESSION, WTSConnectState) == WTSActive`. If either check fails, the tick is skipped. Added 2-min `resume_timeout` — if `WTS_SESSION_UNLOCK` hasn't arrived within 2 min of wake, clear `resume_pending` (some Surfaces skip login).

**Fix 2 — Quit inline ENABLE races with in-flight worker op (GPT)**  
Quit was doing inline SetupAPI on the loop thread even though the worker might still be executing SetupAPI on the same device → concurrent `SetupDiCallClassInstaller` calls.

**Change:** Quit now sends `Cmd::Shutdown` sentinel to worker and calls `join()` with 500 ms timeout. If worker exits cleanly → inline ENABLE+verify. If timeout elapses → proceed to inline ENABLE anyway (worker stuck; inline is safer than leaving kbd disabled) and log the timeout.

**Fix 3 — SP_PROPCHANGE_PARAMS scope/profile unspecified (GPT)**  
Persistence semantics depend on these parameters but they were implicit.

**Change:** Spec now explicit in §4.1: `Scope=DICS_FLAG_GLOBAL`, `HwProfile=0` ("all hardware profiles").

**Fix 4 — Cargo.toml feature-list drift (GPT)**  
`Devices_Bluetooth` was missing from §7 features list but §15 said it was re-added.

**Change:** Added `"Devices_Bluetooth",` to §7 feature list. Updated §15 to say "confirmed in §7."

#### Important (2)

**Fix 5 — No op-id correlation on worker results (GPT)**  
Worker posted results via `PostMessage(WM_APP+3, result)` with no correlation. A stale VerifyState result could arrive after user toggled Active off.

**Change:** Added monotonic `op_id: u64` counter owned by loop thread. Each `Cmd::{Enable|Disable|Verify}` carries an `op_id`. Worker includes it in result payload. Loop tracks `current_generation: u64` — on any state-changing event (toggle, resume, quit), increment it. On result receipt, if `result.op_id < current_generation`, ignore (log at debug). Documented in §4.5.

**Fix 6 — BLE ConnectionStatus staleness honesty (Claude)**  
`ConnectionStatusChanged` fires authoritatively, but reading the property after unclean disconnect (battery dead, out of range) can lag seconds to tens of seconds (L2CAP supervision timeout). The 20 s sanity re-read doesn't help because it reads the same stack-cached value.

**Change:** Added disclosure to §1 "Honest scope" and §4.6 acknowledging the latency. User mitigations: tray 'Active' toggle and `--recover`.

#### Simplifications (3)

**Fix 7 — --recover + hung primary race (Claude)**  
Currently §12 row 2 said to run `kbblock.exe --recover`, but if the hung primary still has the BLE callback alive, it can re-disable the keyboard after `--recover` exits.

**Change:** Expanded row 2 to: (1) run `--recover`, (2) open Task Manager, (3) kill the hung kbblock.exe process. Warns that leaving the hung instance running can re-disable the keyboard.

**Fix 8 — Drop target_poisoned_for_session (Claude)**  
Verify mismatch set a session-scoped poison flag with no UI indication.

**Change:** Replaced with simpler pattern: verify mismatch → ENABLE + set `desired_active=false` + tray balloon "Keyboard disable failed — toggled Active off. Restart app to retry." User toggles Active back on when ready. Removed `target_poisoned_for_session` from all mentions.

## 2026-04-22: Hide Console Window in Release Builds

**Author:** Newman  
**Date:** 2026-04-22  
**Status:** ✅ IMPLEMENTED  
**Category:** UX, Build Configuration

### Context

kbblock.exe is a Windows tray application. When launched by double-clicking from Explorer, it should show **only the tray icon**, not a console window. Rust binaries default to the **console subsystem** on Windows, causing a black terminal window to appear at process start—unacceptable for a GUI-only app.

However, kbblock also has CLI subcommands (`--install-boot-task`, `--uninstall-boot-task`, `--recover`) that need to print output when invoked from a terminal.

### Decision

**Use conditional compilation to select Windows subsystem based on build profile:**

```rust
#![cfg_attr(not(debug_assertions), windows_subsystem = "windows")]
```

- **Release builds** → `windows_subsystem = "windows"` → no console window
- **Debug builds** → `console` subsystem (default) → `println!` works for dev logging

**To preserve CLI output in release builds:**

Call `AttachConsole(ATTACH_PARENT_PROCESS)` at the start of `main()`:

```rust
#[cfg(windows)]
unsafe {
    use windows::Win32::System::Console::{AttachConsole, ATTACH_PARENT_PROCESS};
    let _ = AttachConsole(ATTACH_PARENT_PROCESS);
}
```

This reconnects stdout/stderr to the parent console if one exists (terminal launch). If no parent console exists (Explorer launch), `AttachConsole` fails silently—output is discarded, which is correct for GUI.

### Consequences

**Positive:**
- ✅ Tray launches from Explorer: No console window
- ✅ CLI launches from pwsh/cmd: Output flows to calling terminal
- ✅ Debug builds: Console still available for dev diagnostics
- ✅ Minimal footprint: Single attribute + one API call

**Limitations (expected Windows behavior):**
- ⚠️ UAC-elevated CLI: When `--install-boot-task` triggers elevation, child process detaches from launching console (Windows behavior). Output doesn't appear in parent pwsh, but task is registered (verify via `schtasks /Query`).
- ⚠️ Scheduled task context: `--recover` from SYSTEM task has no parent console, so output is discarded (expected and correct for scheduled tasks).

### Implementation

**Files Changed:**
- `src/main.rs` — Attribute at top (line 1), `AttachConsole` call at start of `main()`
- `Cargo.toml` — `Win32_System_Console` feature (already enabled)

**Build Output:** 378.50 KB (release build, kbblock.exe)

### Testing

✅ Explorer launch: No console window  
✅ CLI from pwsh: Output appears in console  
✅ Debug build: Console allocated (unaffected by cfg_attr)

## Key Decisions

### 1. Event-Token Lifecycle (RAII)

`BleHandle` stores the `EventRegistrationToken` from `ConnectionStatusChanged::add()`. `Drop` impl calls `RemoveConnectionStatusChanged(token)` to unsubscribe. Logs any error but does NOT panic — fail-safe cleanup. This is the Spike 1.6 v4 pattern validated under requireAdministrator (11 clean transitions observed).

### 2. WM_APP Constants Layout

**Decided:** Each module owns its own message constant.

- `WM_BLE_CONNECTION_CHANGED` (WM_APP+1) — defined in ble.rs
- WM_APP+2 (tray), WM_APP+3 (worker result) — deferred to main.rs

Main.rs will define these as either bare constants or in a `pub mod msg` block. Recommendation for main.rs: define `pub const WM_TRAY_EVENT: u32 = WM_APP + 2;` and `pub const WM_WORKER_RESULT: u32 = WM_APP + 3;` directly in main.rs, re-export `ble::WM_BLE_CONNECTION_CHANGED` if needed.

**Rationale:** Decouples module constants. Each module is self-documenting. Main.rs aggregates.

### 3. NotPaired Handling

If `FromBluetoothAddressAsync` returns null device (Nuphy never paired on this machine), `start()` returns `Err(BleError::NotPaired)`. Caller must:

1. Log error loudly (error level, user-visible balloon optional)
2. Do NOT subscribe BLE events
3. Treat `is_connected()` as always-false for the entire session (§4.6 explicit clause)
4. Instruct user to pair Nuphy via Windows Settings → Bluetooth, then restart app

This is fail-safe: internal keyboard stays enabled.

### 4. `is_connected()` Fresh Read (§4.6)

Each call to `is_connected()` invokes `device.ConnectionStatus()` afresh — **NO cache**. This is the v4 state-machine fix (Updated event → IsConnected check, not Removed event). On error:

- Log at warn (HRESULT included)
- Return `false` (fail-safe: treat as not connected → ENABLE path in `apply_policy()`)

Sanity timer (main.rs, 20s cadence) re-reads this; user can also toggle Active via tray.

### 5. HWND Passing Soundness

HWND is Copy. The TypedEventHandler closure captures `hwnd` by value. PostMessage is thread-safe (can be called from any thread per Win32 docs). Message loop is always alive while BleHandle lives (BleHandle is owned by main, dropped before message loop exits). **This is sound.**

No synchronization primitives needed (Arc/Mutex). No lifetime issues. Simple and correct.

### 6. COM Initialization

`start()` calls `CoInitializeEx(None, COINIT_APARTMENTTHREADED)` to initialize COM as STA (Single-Threaded Apartment). This is required for WinRT calls. Main.rs must ensure:

- COM is either uninitialized on the calling thread, OR
- Already initialized as STA (second STA init is idempotent S_FALSE, treated as Ok by windows-rs)

**Do NOT** call `start()` after initializing COM as MTA (`COINIT_MULTITHREADED`) — that will fail.

Spike 1.6 pattern validated this under requireAdministrator. No issues on ARM64.

### 7. Logging Discipline

- **info:** Successful subscribe (includes BD_ADDR), each ConnectionStatusChanged fire (before PostMessage — helps diagnostics / Spike-1.6-style transition counting)
- **warn:** `is_connected()` error (HRESULT included), ConnectionStatusChanged with null sender (defensive)
- **error:** NotPaired (null device from FromBluetoothAddressAsync), Drop unsubscribe failure (HRESULT included)

All log messages prefixed with `"BLE:"` for grep-ability.

## 2026-04-22: Newman — Test 6 Lock-Screen ENABLE Architecture (Modern Standby Fix)

**Date:** 2026-04-22  
**Author:** Newman (Input Hooks Engineer)  
**Status:** Implemented (v0.1 ship-blocker fix)  
**Scope:** Lock-screen keyboard recovery across all lock paths (lid close, Win+L, auto-lock, fast-user-switch)

### Problem

Test 6 (lid close → Modern Standby → lock screen) failed on Surface Laptop 7 ARM64. Internal keyboard was dead at lock screen. Log evidence:
- NO `PBT_APMSUSPEND` fired during lid close
- NO `WTS_SESSION_LOCK` fired
- NO `WTS_SESSION_UNLOCK` in lid-close window
- BLE Disconnected → kbd ENABLE (op_id=9) worked at lock screen when user manually disconnected Nuphy
- Keyboard stayed disabled until OSK login completed

**Root cause:** Modern Standby (S0ix) on Surface Laptop 7 skips `PBT_APMSUSPEND`. We had `WTSRegisterSessionNotification` registered but only handled `WTS_SESSION_UNLOCK` (silently ignored `WTS_SESSION_LOCK`). No power setting notifications registered for lid switch or display state changes.

**Why this matters:** Lock-screen keyboard lockout is the PRIMARY user-safety failure mode. If the user cannot reach the lock screen with their internal keyboard AND their BLE keyboard is disconnected/dead/out-of-range, they are locked out. This is a v0.1 ship-blocker.

### Decision: Three-Layer Defense

All layers converge on: inline ENABLE + set `resume_pending=true`. Unlock path clears flag and re-evaluates policy.

**Layer 1: WTS_SESSION_LOCK (Master Lock-Screen Trigger)**
- Add `WTS_SESSION_LOCK` handler in `handle_session_event()` (src/main.rs ~783)
- On lock → inline ENABLE + set `resume_pending=true` + set `resume_timestamp`
- Log: `"WTS_SESSION_LOCK: inline ENABLE + resume_pending"`
- Delivered even on Modern Standby devices (unlike `PBT_APMSUSPEND`)
- Single most important signal for lock-screen recovery

**Layer 2: Power Setting Notifications (Lid + Display)**
- Register two power setting notifications via `RegisterPowerSettingNotification` during window setup (src/main.rs ~280):
  - `GUID_LIDSWITCH_STATE_CHANGE` (`{BA3E0F4D-B817-4094-A2D1-D56379E6A0F3}`)
  - `GUID_CONSOLE_DISPLAY_STATE` (`{6FE69556-704A-47A0-8F24-C28D936FDA47}`)
- Both registered with `DEVICE_NOTIFY_WINDOW_HANDLE` flag
- Handle `PBT_POWERSETTINGCHANGE` in `handle_power_event()` (now takes `lparam: LPARAM`)
- Parse `POWERBROADCAST_SETTING` structure: if lid state=0 (closed) or display state=0 (off), then inline ENABLE + set `resume_pending=true`
- Log: `"PBT_POWERSETTINGCHANGE: lid closed → inline ENABLE + resume_pending"` or `"display off → ..."`
- Modern Standby devices fire lid/display power setting notifications immediately, often before or instead of `PBT_APMSUSPEND`

**Layer 3: PBT_APMSUSPEND (Legacy Belt-and-Suspenders)**
- Keep existing `PBT_APMSUSPEND` handler (src/main.rs ~769)
- **CHANGE:** Now also sets `resume_pending=true` (was missing before — only called inline_enable)
- Log: `"PBT_APMSUSPEND: inline ENABLE + resume_pending"`
- Backward compatibility with non-Modern-Standby devices
- Belt-and-suspenders: if any of the three layers fires, keyboard recovery succeeds

### Convergence and Idempotency

All three layers call `inline_enable(state)` + set `resume_pending=true`. Safe and idempotent:
- `inline_enable()` is a no-op if keyboard is already enabled
- `resume_pending=true` is a boolean flag (setting it twice does nothing)
- All three paths may fire in quick succession (e.g., lid close → display off → session lock) — each call is independent, no race condition

**Unlock path unchanged:** `WTS_SESSION_UNLOCK` checks `if resume_pending { clear flag + apply_policy() }`. Only one unlock handler, so this runs exactly once per lock/unlock cycle.

### Alternatives Considered (Rejected)

**Alt 1: Only use WTS_SESSION_LOCK, skip power setting notifications**
- Rejected: Test 6 log showed WTS_SESSION_LOCK did NOT fire on lid close. Power setting notifications fire immediately and reliably on Surface. Both needed for defense in depth.

**Alt 2: Use WM_POWERBROADCAST with GUID_ACDC_POWER_SOURCE (AC/battery change) as proxy for lid close**
- Rejected: Not all lid closes trigger AC/battery state change. Lid switch notification is the direct, reliable signal.

**Alt 3: Polling via sanity timer (WTSQuerySessionInformationW every 20s to detect lock state change)**
- Rejected: Test 6 requires immediate keyboard recovery on lock (not up to 20s delay). Event-driven is the right model.

### Testing

**Test 6 retry acceptance criteria:**
- Internal keyboard MUST be enabled at lock screen after lid close
- Log MUST show at least one of: `WTS_SESSION_LOCK`, `PBT_POWERSETTINGCHANGE: lid closed`, or `PBT_POWERSETTINGCHANGE: display off` during lid-close window
- After OSK login, internal keyboard MUST be disabled again (if Nuphy still connected)

**Log artifacts to verify:**
- `"WTS_SESSION_LOCK: inline ENABLE + resume_pending"` (if lock notification fires), OR
- `"PBT_POWERSETTINGCHANGE: lid closed → inline ENABLE + resume_pending"` (if lid notification fires first), OR
- `"PBT_POWERSETTINGCHANGE: display off → inline ENABLE + resume_pending"` (if display-off fires)
- `"WTS_SESSION_UNLOCK: clearing resume_pending, calling apply_policy()"` (on OSK login)
- `"Policy: Nuphy connected, predicate Ok → Disable (op_id=N)"` (after unlock, if Nuphy still connected)

**If none of the three triggers fire:** Check kbblock.exe is the rebuilt binary (v0.1 with this fix). If still no triggers, escalate to Brady — indicates deeper Windows configuration issue (e.g., session notifications disabled, power policy override).

### Impact on Other Tests

**Tests 1-5:** No change expected (all passed before this fix). New handlers are additive, not replacements. Existing BLE connect/disconnect path unchanged.

**Test 7-12:** Lock-screen recovery now works reliably. Tests that involve lid close, display off, or Win+L will now show correct keyboard state at lock screen.

### Cargo.toml Feature Requirements

**Verified:** `Win32_System_Power` and `Win32_UI_WindowsAndMessaging` already present in Cargo.toml features list. No new dependencies required. APIs imported via existing wildcard imports.

### Future Work (Out of Scope for v0.1)

1. **Unregister power setting notifications on exit:** Currently we register but never unregister. Not a leak (Windows cleans up on process exit), but better hygiene would be to unregister in shutdown cleanup. Deferred to v0.2+.
2. **Tooltip indicator for resume_pending state:** Consider adding transient `"Active: On (lock screen)"` tooltip when `resume_pending=true`. Deferred pending user feedback.
3. **RDP lock-screen test:** Test 6 covers local lid close. RDP disconnect (when lock is configured) should also fire WTS_SESSION_LOCK, but not verified on-device. Add to Test 13+ for v0.2.

### Takeaway for Future Agents

**Modern Standby devices (Surface, most post-2015 laptops) skip classic suspend events.** The reliable triggers for lock-screen keyboard recovery are:
1. **WTS_SESSION_LOCK** — master lock-screen signal (all lock paths)
2. **GUID_LIDSWITCH_STATE_CHANGE** — physical lid close (fires on Surface even when PBT_APMSUSPEND doesn't)
3. **GUID_CONSOLE_DISPLAY_STATE** — display off (covers screen blank without lock)

**Always register all three** via `WTSRegisterSessionNotification` + `RegisterPowerSettingNotification` for lock-screen reliability on modern Windows laptops. PBT_APMSUSPEND is legacy but still useful for classic S3 suspend on older/desktop hardware — keep it as belt-and-suspenders.

**Do NOT assume PBT_APMSUSPEND fires on Modern Standby.** It doesn't. This is a known Windows quirk (S0ix low-power idle != S3 suspend). See Microsoft Docs: Modern Standby for background.

## 2026-04-22: Collapse Helper EXEs into Single kbblock.exe with Self-Elevation (SUPERSEDES 2026-04-22 Helper EXE Design)

**By:** Newman (implementation + pivot execution)  
**Date:** 2026-04-22T13:30:00Z  
**Status:** Implemented. Awaiting Brady's runtime testing.  
**Supersedes:** `.squad/decisions/inbox/newman-task-exes.md` (the three-EXE design with separate add/remove helpers)

### Context

Two days ago, we shipped `add-scheduled-task.exe` and `remove-scheduled-task.exe` as separate UAC-manifested binaries. Brady ran the UX past real users and collected critical feedback: **three EXEs in a release zip is confusing**. Expected: single `kbblock.exe` with everything inside (tray menu, CLI flags, self-elevation where needed). Original two-helper design was technically correct but user-unfriendly.

### Decision: Consolidate to Single EXE with Self-Elevation

Fold both helpers into `kbblock.exe`:

1. **CLI subcommands:** `kbblock.exe --install-boot-task` and `kbblock.exe --uninstall-boot-task` replace the two helper EXEs.
2. **Tray menu:** Add "Boot recovery task" check-item alongside "Active", "Start at login", and "Quit" (four items total).
3. **Manifest:** `kbblock.exe`'s manifest stays at `asInvoker` (Brady's hard requirement — tray must launch without UAC). Self-elevation via `ShellExecuteExW` with `runas` verb when user invokes boot-task toggle unprivileged.
4. **Stale-path warnings:** MessageBoxW (modal, unambiguous) fires at startup if HKCU Run or Task Scheduler paths drift from current_exe(). User re-toggles to refresh paths.
5. **Fallback:** PowerShell scripts (register-recovery-task.ps1, unregister-recovery-task.ps1) demoted to tertiary fallback for power users.

### Self-Elevation Pattern (Key Technical Detail)

```rust
// When boot-task toggle invoked unprivileged:
let mut info = SHELLEXECUTEINFOW {
    cbSize: size_of::<SHELLEXECUTEINFOW>() as u32,
    fMask: SEE_MASK_NOCLOSEPROCESS,
    lpVerb: w!("runas"),

    lpParameters: <"--install-boot-task".wide()>,  // or --uninstall-boot-task
    nShow: SW_HIDE.0 as i32,

GetExitCodeProcess(info.hProcess, &mut code)?;

**When elevated:** Call boot_task::install/uninstall directly (no re-prompt).  
**When not elevated:** Relaunch via ShellExecuteExW with "runas" verb.  
**On UAC deny:** ShellExecuteExW fails; re-sync menu check state to pre-click value (no crash).  
**Threading:** Worker thread processes tray menu events; WaitForSingleObject blocks worker (acceptable for one-shot operation, not UI message pump).

### Stale-Path Detection & Warning Mechanism

**Problem:** HKCU Run and Task Scheduler both hard-code an absolute path to kbblock.exe. If user moves the binary, paths drift silently.

**Solution:**
1. At startup, check registered paths against current_exe() via paths_equal() (canonicalizes + lowercases).

- Silent re-pointing: Presumptuous — user may have intentionally pointed elsewhere

# Decision: COM Apartment-Mode Tolerance Policy

**Date:** 2026-04-22  
**Context:** Turn 2 of tray regression saga. Non-admin boot_task.rs cold-start hits `RPC_E_CHANGED_MODE` (0x80010106) when Task Scheduler COM init conflicts with existing STA mode from BLE layer.

## Policy
When calling COM APIs (Task Scheduler, etc.) from a process that has already initialized COM in STA mode elsewhere (e.g., Bluetooth LE device enumeration), **tolerate `RPC_E_CHANGED_MODE` (0x80010106) as success**. This is a legitimate state, not a failure.

**Implementation:**
```rust
// In boot_task.rs or any COM call site
let result = CoCreateInstance(/* ... */);
if result == RPC_E_CHANGED_MODE {
    // Already in a different apartment mode. Treat as OK for read-only operations.
    return Ok(());
}
```

**Corollary:** Only call `CoUninitialize` if `CoInitialize` or `CoInitializeEx` actually succeeded in THIS module. Do not blindly uninitialize.

## Rationale
- **Symptom:** `RPC_E_CHANGED_MODE` fires when trying to call COM APIs from a thread that has COM already initialized in a conflicting mode (e.g., main thread is STA, we try to call Task Scheduler's methods expecting MTA).
- **Root cause:** Bluetooth device enumeration (BLE) initializes COM in STA. Subsequent Task Scheduler calls (boot_task.rs) hit the apartment conflict.
- **Fix:** Rather than treating this as a fatal error, recognize it as a benign conflict when all we need is to *query* Task Scheduler state. Read-only COM operations work fine in this state; only mutation might fail (unlikely in our use case).

## Files Affected
- `src/boot_task.rs` — apply this tolerance in CoCreateInstance + Task Scheduler query paths

# Decision: Startup Must Not Bypass Tray Icon

**Date:** 2026-04-22  
**Context:** Tray regression saga. Root cause: non-admin cold-start ENABLE failure called `std::process::exit(1)` before tray init. User saw no feedback (no tray, no dialog).

## Rule
**NO startup failure** (cold-start ENABLE, autostart registry query, boot task query, permission check, COM conflict, etc.) **may short-circuit tray init** via:
- `std::process::exit()` / `panic!()`
- Early return out of main
- Unwrap on recoverable errors
- Any path that prevents `create_tray()` or `event_loop()`

## Rationale
Under `windows_subsystem = "windows"`, the **tray icon is the user's only feedback channel**. There is no console window, no stdout, no stderr. If startup fails silently, the user is left staring at nothing, unable to recover or quit.

## Implementation Pattern
```rust
// main.rs
fn main() {
    init_logging();  // Non-fatal
    check_elevation();  // Non-fatal; note in state if failed

    if let Err(e) = cold_start_enable() {
        log_error!("cold_start ENABLE failed: {}", e);
        // DO NOT EXIT HERE
        // Instead, remember failure + show in tray
    }

    // ALWAYS build tray, even if cold-start failed
    match create_tray(is_admin) {
        Ok(tray_handle) => run_event_loop(tray_handle),
        Err(e) => {
            MessageBoxW(/* Critical error, no recovery */);
            exit(1);
        }
    }
}
```

**Degradation:** If not elevated, show needs-admin tray (menu disabled, tooltip explains). If a startup operation fails, log it + show balloon notification. But **tray always renders**.

## Files Affected
- `src/main.rs` — enforce this pattern in all startup paths
- `src/boot_task.rs` — no `exit()` calls
- `src/autostart.rs` — no `exit()` calls

## Exception
Only `create_tray()` itself may fail catastrophically (if it fails, there's no recovery path). All prior startup logic must be tolerant.

# Decision: Elevation Detection Is Baked In

**Date:** 2026-04-22  
**Context:** Tray regression saga Turn 3. Non-admin launch left user with no feedback because startup called `exit(1)` before tray rendered.

## Pattern
`check_elevation()` via `GetTokenInformation(TokenElevation)` is now permanent in `src/main.rs`. Call it early.

### Implementation
```rust
use windows::Win32::Security::{GetTokenInformation, TokenElevation};
use windows::Win32::Foundation::GetCurrentProcess;
use windows::Win32::Security::GetCurrentProcessToken;

fn check_elevation() -> bool {
    unsafe {
        let token = match GetCurrentProcessToken() {
            Ok(t) => t,
            Err(_) => return false,
        };

        let mut elevation = TOKEN_ELEVATION::default();
        let mut size = std::mem::size_of::<TOKEN_ELEVATION>() as u32;

        match GetTokenInformation(
            token,
            TokenElevation,
            Some(&mut elevation as *mut _ as *mut _),
            size,
            &mut size,
        ) {
            Ok(_) => elevation.TokenIsElevated != 0,
            Err(_) => false,
        }
    }
}
```

### Usage Gate
Use `check_elevation()` return value as a gate for:
- **HKLM\SYSTEM\CurrentControlSet** writes (HID device control, service config)
- **Firewall rule creation** (if needed)
- **Boot task creation** (UAC-protected registry paths)

When not elevated:
- Log: "Not running as administrator. Some features require elevation."
- Show tray with **needs-admin mode**: menu items disabled, labels include "(needs admin)", tooltip explains
- Never **attempt** privileged operations; wait for user to launch elevated or click menu item to self-elevate

# Decision: Diagnostic Instrumentation Is Permanent

**Date:** 2026-04-22  
**Context:** Tray regression saga. Turn 2 added panic hook + phase logging. Turn 3 Coordinator used phase.log to decode root cause in minutes. **Never remove this.**

## What's Baked In

### 1. Panic Hook
**File:** `%LOCALAPPDATA%\kbblock\panic.log`

```rust
// In main.rs startup
std::panic::set_hook(Box::new(|panic_info| {
    if let Ok(path) = std::env::var("LOCALAPPDATA") {
        let log_path = PathBuf::from(path)
            .join("kbblock")
            .join("panic.log");
        if let Ok(mut file) = std::fs::OpenOptions::new()
            .create(true)
            .append(true)
            .open(&log_path)
        {
            let _ = writeln!(file, "[{}] PANIC: {}", 
                chrono::Local::now().format("%Y-%m-%d %H:%M:%S"),
                panic_info
            );
        }
    }
}));
```

**Purpose:** If the app panics, we have a permanent record. Even if crash-recovery deletes temp logs, panic.log persists.

### 2. Phase Tracing
**File:** `phase.log` (same dir as kbblock.log)

```rust
// Add phase breadcrumbs at key startup points:
log_phase!("cold_start_start");
if let Err(e) = cold_start_enable() {
    log_phase!("cold_start_failed");
}
if !is_admin {
    log_phase!("cold_start_skipped_not_elevated");
} else {
    log_phase!("cold_start_done");
}
log_phase!("tray_init_start");
log_phase!("calling_create_tray");  // or calling_create_tray_needs_admin
log_phase!("tray_menu_built");
log_phase!("event_loop_start");
```

**Purpose:** Trace startup flow without verbose logging spam. Coordinator can decode a sequence of 5–10 breadcrumbs in seconds to pinpoint where a failure occurred.

## Why They Saved The Day
- **Turn 1:** Misdiagnosed as panic → looked at panic.log → empty (proved not a panic)
- **Turn 2:** Added instrumentation, but log showed full startup → no tray → silent exit
- **Turn 3:** Phase log showed `logging_init_done` → (nothing) → exit. Coordinator: "startup failed before tray init, and there's no panic? Must be an early `exit(1)` call."

Without these logs, Turn 3 would have been blind debugging.

## Maintenance Rule
- **Never remove** panic hook or phase tracing infrastructure
- **When adding new startup logic**, add corresponding `log_phase!()` breadcrumbs
- **When deleting startup logic**, remove its breadcrumbs
- Keep phase breadcrumbs terse (3–5 words); they're not for humans to read line-by-line, but for decoding a sequence

- `src/logging.rs` — log_phase! macro + file append logic (non-blocking)

## 2026-04-22: Silent Logon Elevation via Scheduled Task

**Date:** 2026-04-22  
**Author:** Newman (Backend/Systems Dev)  
**Status:** Implementation Complete  
**Affects:** v0.1.2 autostart UX  

### Problem

v0.1.1 embedded `requireAdministrator` in manifest, which fixed cold-start ENABLE but broke autostart UX:

- At every Windows logon, Explorer launches kbblock.exe from HKCU Run registry
- Because the exe has `requireAdministrator` manifest, **UAC consent dialog appears EVERY login**
- Brady's requirement: silent logon (no UAC prompt)

### Decision

Replace the HKCU Run-key autostart mechanism with a **Task Scheduler logon-trigger task** that uses `<RunLevel>HighestAvailable</RunLevel>` to pre-elevate the token at logon without prompting UAC.

### Task Configuration

Task name: `kbblock-logon` (under root `\` folder, consistent with `kbblock-boot-recover` naming)

Key XML elements:
- `<LogonTrigger><UserId>` scoped to current interactive user (only triggers for the user who enabled it)

- `<ExecutionTimeLimit>PT0S</ExecutionTimeLimit>` — no timeout (kbblock is long-running)

## 2026-04-22: Tray Icon Regression Fix — Logging Panic in windows_subsystem Mode

**Author:** Newman  
**Date:** 2026-04-22  
**Status:** ✅ FIXED  
**Category:** Bug Fix, Critical Regression

### Context

After implementing the hide-console feature (adding `#![cfg_attr(not(debug_assertions), windows_subsystem = "windows")]` + `AttachConsole(ATTACH_PARENT_PROCESS)`), Brady reported:

✅ Console window no longer pops up (fix worked)  
❌ Tray icon shows "error state" with no menu items visible  
❌ Right-click "Quit" does not work — menu unresponsive

Critical regression from the `windows_subsystem = "windows"` change.

### Root Cause

The `init_logging()` function used `.expect("Failed to open log file")` when opening the log file. When the log file couldn't be opened for any reason (permissions, disk full, etc.), this would **panic**.

**Critical interaction with `windows_subsystem = "windows"`:**
- With `windows_subsystem = "windows"`, the app has no console window
- When a panic occurs, there's **no stderr** to write the panic message to
- The app terminates **silently** during startup, before the tray icon is fully initialized
- This leaves the tray in an "error state" with no menu items

**Startup sequence that caused the bug:**
1. `main()` starts
2. `AttachConsole(ATTACH_PARENT_PROCESS)` is called — returns false when launched from Explorer (no parent console)
3. `init_logging()` is called
4. Log file open fails → `.expect()` panics
5. With no console, the panic is invisible
6. App terminates before tray initialization completes
7. User sees broken tray icon

### The Fix

Changed `init_logging()` to handle log file errors gracefully instead of panicking:

```rust
let log_file = match std::fs::OpenOptions::new()
    .create(true)
    .append(true)
    .open(&log_path) {
    Ok(f) => f,
    Err(_) => return, // Silently skip logging if file can't be opened
};
```

**Key changes:**
- Replaced `.expect("Failed to open log file")` with `match` statement
- Returns early if log file can't be opened instead of panicking
- App continues to run even if logging can't be initialized

### Why This Works

1. **Fail-soft logging:** If logging can't be initialized, the app continues without logging rather than crashing
2. **No silent panics:** Removes the `.expect()` that was causing invisible panics in `windows_subsystem` mode
3. **Preserves functionality:** All app features (tray, menu, keyboard blocking) work even if logging is disabled
4. **Appropriate for tray app:** A background tray app should be resilient to non-critical failures like logging

### Build Output

- **Binary:** `output\kbblock.exe`
- **Size:** 379 KB
- **Target:** aarch64-pc-windows-msvc
- **Build profile:** Release with `windows_subsystem = "windows"`

### Testing Required

✅ Explorer launch: Tray icon appears with full menu  
✅ Right-click menu: All items responsive  
✅ "Active" checkbox: Works  
✅ "Quit" menu item: Cleanly exits the app  
✅ CLI launch from pwsh: Output still flows to console  
✅ No console window on Explorer launch

### Lessons Learned

**Critical lesson for windows_subsystem apps:**

When using `#![cfg_attr(not(debug_assertions), windows_subsystem = "windows")]`:
- **All panics are invisible** — there's no console to display panic messages
- **Never use `.expect()`, `.unwrap()`, or `panic!()` on initialization code** that could fail
- **Use graceful error handling** with `match`, `if let Ok()`, or `.ok()` for non-critical operations
- **Test launch scenarios:** Explorer launch vs. terminal launch — panics are visible in terminal but not Explorer
- **Consider fail-soft patterns:** Apps should continue running even if non-critical features (like logging) fail
